# Implementation Plan: DDD Architecture Refactoring

**Branch**: `006-ddd-refactor` | **Date**: 2025-01-15 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/006-ddd-refactor/spec.md`

## Summary

This plan outlines the architectural refactoring to adopt Domain-Driven Design (DDD) patterns using Ardalis.SharedKernel, eliminate duplicate classes, and remove the CLI application. The primary goals are:

1. **Simplify application structure** by removing the CLI project and consolidating on the web dashboard
2. **Establish single source of truth** by eliminating duplicate entity definitions (RiskSettings, EquityPoint)
3. **Implement DDD patterns** using SharedKernel base classes for entities, value objects, aggregates, and domain events
4. **Improve maintainability** through consistent domain modeling and reduced code duplication

Technical approach involves three parallel workstreams: CLI removal (delete project + update references), duplicate elimination (merge classes + update usages), and DDD adoption (add SharedKernel + refactor entities to extend base classes).

## Risk Mitigation

### Risk 1: EF Core Compatibility with SharedKernel
**Likelihood**: Low | **Impact**: High
**Mitigation**: Phase 0 research confirmed compatibility. EntityBase uses standard properties that EF Core handles naturally.

### Risk 2: Breaking Changes in Entity Equality
**Likelihood**: Medium | **Impact**: Medium
**Mitigation**: EntityBase provides value-based equality. Existing tests will catch any behavioral changes. Add explicit equality tests.

### Risk 3: Domain Event Performance Overhead
**Likelihood**: Low | **Impact**: Low
**Mitigation**: Events dispatched after transaction commit. Async handling prevents blocking. Monitor with existing APM.

### Risk 4: Test Failures After Refactoring
**Likelihood**: Medium | **Impact**: Medium
**Mitigation**: Refactor incrementally, run tests after each entity conversion. Keep test logic unchanged, only update type references.
